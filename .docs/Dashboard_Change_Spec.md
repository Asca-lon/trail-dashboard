# 개요

이 문서는 2026-07-10 기준 대시보드 프론트엔드 개선사항을 다른 개발자가 빠르게 이해하고 이어서 개발할 수 있도록 정리한 인수인계 문서입니다.

대상 화면은 `frontend/dashboard.html`이며 사이드바 변경은 범위에서 제외합니다.

현재 작업에는 다음 제약이 적용됩니다.

- `frontend/`와 프론트엔드 관련 Markdown 문서만 수정합니다.
- `backend/`, `mock/`, DB, 수집·분석 코드는 수정하지 않습니다.
- 기존 Mock 데이터는 절대 변경하지 않습니다.
- Mock에 없는 값은 임의로 생성하지 않습니다.

# 구현 목적

- 발효된 모든 기상특보와 영향 지역을 누락 없이 표시합니다.
- 대시보드 필터 상태와 향후 API 쿼리 형식을 통일합니다.
- 데이터에 존재하는 필터 선택지만 동적으로 제공합니다.
- 대시보드, 역 상세, 구간 상세의 노선 기준을 경부선 고속철도로 통일합니다.
- 요약 카드와 위험도 배지의 계산 기준을 명확하게 만듭니다.
- 정확한 데이터, 추정 데이터, 제공되지 않은 데이터를 사용자에게 구분해 표시합니다.

# 구현 내용

## 1. 다중 기상특보 배너

기존에는 `active[0]`의 첫 번째 특보만 배지로 표시했습니다. 현재는 다음 복합 키로 발효 특보를 그룹화합니다.

```text
alert_type + alert_level
```

예:

```text
호우 경보 · 대전
폭염 주의보 · 동대구
```

동일한 특보가 여러 지역에 발효되면 첫 번째 지역만 표시하고 나머지는 다음과 같이 요약합니다.

```text
호우 경보 · 대전 외 3개 지역
```

`외 3개 지역`에 마우스를 올리거나 키보드 포커스를 이동하면 숨은 지역 목록을 표시합니다. 마우스 이탈, 포커스 이탈, `Escape` 입력 시 목록을 닫습니다. 터치 환경에서는 버튼으로 열고 닫을 수 있습니다.

영향 가능 구간 수는 각 특보의 `affected[]` 중 `type === "segment"`인 항목 수입니다.

## 2. 프론트엔드 공통 필터 규격

대시보드 필터 상태는 다음 형태로 통일했습니다.

```javascript
{
  line: "경부선",
  alertType: "all",
  alertLevel: "all",
  trainType: "all"
}
```

화면의 `전체`, 빈 문자열은 내부적으로 `all`로 정규화합니다.

향후 실제 API 요청에 사용할 쿼리 이름은 다음과 같습니다.

| 화면 필드 | API 쿼리 필드 |
|---|---|
| `line` | `line` |
| `alertType` | `alert_type` |
| `alertLevel` | `alert_level` |
| `trainType` | `train_type` |

생성 예시:

```text
line=경부선&alert_type=all&alert_level=all&train_type=all
```

현재는 이 쿼리를 실제 API로 보내지 않고 `dashboardState.filterQuery`에 보관합니다.

## 3. 필터 선택지 동적 생성

HTML에는 초기 상태인 `전체` option만 둡니다. 데이터 로딩 후 JavaScript가 다음 필드를 수집해 선택지를 생성합니다.

- 특보 종류: `alert_type`, `alert_types`
- 특보 등급: `alert_level`, `alert_levels`
- 열차 종류: `train_type`, `train_types`

중복값과 빈 문자열은 제거합니다. 특보 등급은 `주의보 → 경보`, 열차 종류는 `KTX → 새마을 → 무궁화` 순서를 우선합니다.

현재 Mock에는 열차 종류 데이터가 없으므로 열차 종류 필터에는 `전체`만 표시하고 비활성화합니다.

## 4. 경부선 고속철도 노선도 기준

대시보드 노선도는 다음 10개 정차역을 기준으로 합니다.

```text
서울 → 광명 → 천안아산 → 오송 → 대전
→ 김천(구미) → 동대구 → 경주 → 울산 → 부산
```

각 기준 역은 `stationId`와 표시명을 함께 가집니다. 인접한 역 사이의 9개 구간만 고속철도 구간으로 판단합니다.

Mock 데이터는 일반 경부선 역을 포함하므로 이름과 구간이 정확히 일치할 때만 취약도를 결합합니다. 천안 데이터를 천안아산에 사용하는 것처럼 다른 역의 데이터를 임의로 대체하지 않습니다.

현재 Mock 일치 현황:

| 구분 | 기준 개수 | 일치 개수 |
|---|---:|---:|
| 고속철도 역 | 10 | 5 |
| 고속철도 인접 구간 | 9 | 2 |

일치하는 역은 서울, 대전, 김천(구미), 동대구, 부산입니다. 일치하지 않는 역과 구간은 데이터 없음으로 표시합니다.

## 5. 요약 카드 지표

모든 카드의 노선 범위는 경부선 고속철도입니다.

### 영향 가능 구간

현재 발효 특보의 영향 대상 중 고속철도 인접 구간에 해당하는 고유 구간 수입니다. 정방향과 역방향은 같은 구간으로 계산합니다.

현재 Mock 결과:

```text
1개
```

### 운행 지연 예상 열차

현재 데이터에는 예상 지연 열차 수, 열차 번호, 운행일이 없습니다. 따라서 다음 상태를 표시합니다.

```text
-편
집계 데이터 없음
```

향후 `expected_delayed_train_count`가 있으면 직접 사용합니다. `expected_delayed_trains[]`만 있으면 `run_date + train_no`로 중복 제거합니다.

### 평균 신규 지연(예상)

고속철도와 정확히 일치하는 취약 구간의 `avg_delay_incr`를 `sample_n`으로 가중평균합니다.

```text
Σ(avg_delay_incr × sample_n) / Σ(sample_n)
```

현재 Mock 계산:

```text
(14.6 × 37 + 6.1 × 33) / (37 + 33) = 10.6분
```

### 취약도 높음 구간

고속철도 인접 구간 중 히트맵의 취약도 점수 판정 결과가 `높음`인 구간 수입니다.

현재 Mock 결과:

```text
1개
```

## 6. 위험도 판정 기준

등급 이름과 색상은 공통이지만 지표 단위에 따라 임계값을 구분합니다.

| 지표 | 관심 | 주의 | 높음 |
|---|---:|---:|---:|
| 취약도 점수 `vuln` | 0.50 미만 | 0.50 이상 | 0.70 이상 |
| 신규 지연 `avg_delay_incr` | 7분 미만 | 7분 이상 | 12분 이상 |
| 역 지연율 `delay_rate` | 25% 미만 | 25% 이상 | 40% 이상 |

숫자가 아니거나 값이 없으면 `데이터 없음`입니다.

적용 영역:

| 화면 | 판정 지표 |
|---|---|
| 노선도 역·구간 | 취약도 점수 |
| 구간 요약 및 고위험 카드 | 취약도 점수 |
| 취약 구간 TOP 5 | 신규 지연 |
| 취약 역 TOP 5 | 역 지연율 |
| 우선 점검 대상 | 신규 지연 |

이 임계값은 기존 프론트엔드 문서에 기록된 표시 정책입니다. `CONTRACT.md`에는 지표의 의미와 단위만 정의돼 있고 위험도 임계값은 없습니다.

## 7. 지연 건수와 선택적 집계 데이터

취약 역 테이블은 기존에 `sample_n`을 지연 건수로 잘못 표시했습니다. 현재는 다음 우선순위를 사용합니다.

```javascript
delay_count ?? Math.round(sample_n * delay_rate)
```

정확한 `delay_count`가 없으면 반올림된 추정값이므로 `약 N건`으로 표시합니다.

현재 Mock 결과:

| 역 | 표시값 |
|---|---:|
| 대전 | 약 27건 |
| 동대구 | 약 21건 |
| 김천(구미) | 약 12건 |
| 수원 | 약 10건 |
| 밀양 | 약 1건 |

전일 비교는 현재값과 이전값이 모두 있을 때만 계산합니다.

| 조건 | 표시 |
|---|---|
| 증가 | `전일 대비 +N ↑` |
| 감소 | `전일 대비 -N ↓` |
| 동일 | `전일과 동일` |
| 이전값 없음 | `전일 비교 데이터 없음` |

# 코드 설명

주요 함수와 역할은 다음과 같습니다.

| 함수·설정 | 역할 |
|---|---|
| `DEFAULT_DASHBOARD_FILTERS` | 공통 필터 기본 상태 |
| `RISK_THRESHOLDS` | 지표별 위험도 임계값 |
| `GYEONGBU_HIGH_SPEED_STATIONS` | 고속철도 기준 역 ID와 순서 |
| `groupActiveAlerts()` | 특보 종류·등급별 그룹화 |
| `renderDashboardFilterOptions()` | 동적 필터 option 생성 |
| `isHighSpeedSegment()` | 고속철도 인접 구간 판정 |
| `getWeightedAverageHighSpeedDelay()` | 표본 가중평균 계산 |
| `getRiskLevel()` | 공통 위험도 판정 |
| `formatDayOverDayComparison()` | 전일 대비 표시 생성 |
| `getExpectedDelayedTrainCount()` | 선택적 예상 열차 집계 처리 |

실행 흐름:

1. 6개 Mock JSON을 병렬로 요청합니다.
2. 필터 선택지를 로드된 데이터에서 생성합니다.
3. 원본 데이터와 공통 필터 상태를 `dashboardState`에 저장합니다.
4. 적용하기 버튼을 누르면 메모리의 데이터를 필터링합니다.
5. 특보 배너, 요약 카드, 노선도, 순위표, 우선 점검 대상을 다시 렌더링합니다.
6. 요청 중 하나라도 실패하면 현재는 전체 빈 상태를 렌더링합니다.

장점:

- 화면의 데이터 범위와 계산식이 명시적입니다.
- Mock에 없는 값을 임의로 표시하지 않습니다.
- 추후 선택적 집계 필드가 추가되면 기존 렌더링 구조를 재사용할 수 있습니다.

단점:

- 아직 실제 API를 호출하지 않습니다.
- 요청 하나가 실패하면 전체 대시보드가 실패합니다.
- 현재 Mock과 고속철도 기준이 달라 데이터 없음 구간이 많습니다.
- 예상 지연 열차와 전일 비교는 실제 데이터가 제공될 때까지 표시할 수 없습니다.

다른 구현 방법:

- 요약 전용 API에서 현재값, 전일값, 단위, 기준 시간을 함께 제공합니다.
- 필터 메타데이터 API에서 허용값과 표시 순서를 제공합니다.
- 노선 메타데이터 API에서 역 ID, 표시명, 정차 순서, 구간 ID를 제공합니다.
- 위험도 등급과 정책 버전을 분석 API가 직접 반환합니다.

실무에서는 프론트엔드가 운영 지표를 직접 조합하기보다 백엔드의 버전 관리된 요약·정책 데이터를 표시하는 구조가 안전합니다.

현재 프론트엔드는 요청 실패를 콘솔에 기록하고 전체 빈 상태를 표시합니다. 사용자용 오류 메시지와 재시도 기능은 아직 구현하지 않았습니다.

# Database

이번 작업에서는 Database를 변경하지 않았습니다.

향후 필요한 데이터:

- 열차 종류별 취약도 집계
- 정확한 역별 `delay_count`
- 예상 지연 열차 고유 식별자
- 카드별 전일값

# 주의사항

## 수정 금지 범위

- `mock/*.json`
- `backend/`
- DB 스키마
- 수집기와 분석기

## 현재 알려진 제약

- 열차 종류 필터는 데이터가 없어 비활성화됩니다.
- 히트맵 Mock에는 특보 종류와 등급이 없어 해당 조건을 적용할 수 없습니다.
- 특보 배너의 영향 구간 수는 안정적인 `segment_id`가 없어 특보 그룹 내부에서 중복될 수 있습니다.
- 역 지연 건수의 `약 N건`은 정확한 원본 건수가 아닙니다.
- 선택적 요약 필드는 프론트엔드 준비 상태이며 현재 API 계약에 확정된 필드는 아닙니다.

## 남은 작업

1. 표본 10건 미만 `근거 부족` 표시 — 사용자 요청으로 보류
2. `전체 보기` 링크 동작 결정 및 구현 — 사용자 요청으로 보류
3. Mock 호출을 실제 API 호출로 전환
4. 로딩 상태 구현
5. API별 부분 오류 처리
6. 사용자 오류 메시지와 재시도 버튼
7. 고속철도 데이터 누락 정책 최종 검토
